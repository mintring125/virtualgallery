# 교실 가상 갤러리 서비스 계획

## 1. 프로젝트 목표

교실에서 만든 미술 작품 사진이나 국어 글을 학생별로 업로드하고,  
사용자들이 아바타로 가상 전시 공간을 돌아다니며 작품을 함께 보고 댓글을 남길 수 있는 웹 서비스를 만든다.

참고 경험은 `spatial.io`와 비슷하지만, 목표는 더 단순하고 교육 현장에 맞는 구조다.

핵심 가치는 아래 4가지다.

- 작품을 "한 장씩" 쉽게 올릴 수 있어야 한다.
- 학생 이름 또는 번호가 포함된 파일명을 이용해 여러 작품을 한 번에 등록할 수 있어야 한다.
- 학생과 교사가 같은 공간에서 함께 감상할 수 있어야 한다.
- 작품마다 댓글과 반응이 남아야 한다.
- 복잡한 조작 없이 PC와 태블릿에서 바로 열려야 한다.

## 2. 사용자

### 교사

- 전시 공간 생성
- 작품 업로드 및 배치
- 공개 범위 설정
- 부적절한 댓글 관리
- PC에서 키보드 중심으로 갤러리 운영

### 학생

- 아바타로 갤러리 입장
- 작품 감상
- 작품별 댓글 작성
- 친구들과 동시에 관람
- 태블릿에서 화면 다이얼패드로 아바타 이동

### 보호자 또는 외부 관람자

- 링크로 입장
- 제한된 권한으로 감상
- 댓글은 허용 여부에 따라 선택적으로 사용

## 3. MVP 범위

첫 버전은 아래 기능에 집중한다.

### 반드시 포함

- 갤러리 1개 생성
- 작품 카드 업로드
  - 이미지
  - 글 작품
  - 제목, 학생 이름, 설명
- 다중 파일 일괄 업로드
  - 학생 이름/번호가 포함된 파일명 자동 분석
  - 업로드 전 미리보기 및 오류 표시
  - 일치하지 않는 항목 수동 수정
- 가상 전시 공간에서 아바타 이동
- 다른 접속자의 위치 표시
- 작품 클릭 시 상세 보기
- 작품별 댓글
- 간단한 관리자 화면
  - 작품 추가/수정/삭제
  - 댓글 삭제

### 있으면 좋은 기능

- 좋아요 또는 스티커 반응
- 음성 채팅 대신 텍스트 채팅
- 갤러리 테마 변경
- 발표 모드
  - 교사가 특정 작품으로 전체 시점 유도

### 제외

- VR 헤드셋 지원
- 3D 모델 업로드
- 복잡한 월드 제작 기능
- 정교한 아바타 커스터마이징

## 4. 권장 기술 방향

현재 목표 기준으로는 "완전한 3D 메타버스"보다 "가벼운 3D 또는 2.5D 전시 공간"이 적합하다.

### 프론트엔드

- `Next.js`
- `React`
- `TypeScript`
- `react-three-fiber` 또는 `three.js`

### 백엔드

- `Next.js API` 또는 별도 `Node.js` 서버
- 실시간 동기화용 `Socket.IO` 또는 `Colyseus`

### 데이터 저장

- `Supabase` 추천
  - 인증
  - DB
  - 파일 업로드
  - 실시간 기능 일부 활용 가능

### 초기 권장 조합

가장 현실적인 1차 조합:

- `Next.js + TypeScript`
- `react-three-fiber`
- `Supabase`
- `Socket.IO`

이 조합이면 구현 난이도와 확장성의 균형이 좋다.

## 5. 화면 구성

### 1. 랜딩 페이지

- 서비스 소개
- 갤러리 입장 버튼
- 교사용 관리 버튼

### 2. 갤러리 입장 페이지

- 이름 입력
- 역할 선택
  - 학생
  - 교사
  - 관람자
- 아바타 색상 또는 기본 캐릭터 선택

### 3. 가상 갤러리 화면

- 중앙: 전시 공간
- 좌측 또는 하단: 온라인 사용자 목록
- 우측: 선택 작품 정보 / 댓글 패널
- 상단: 현재 갤러리 이름, 나가기 버튼
- 태블릿에서는 좌측 하단에 고정 다이얼패드 배치
- 태블릿에서는 작품 정보와 댓글 패널을 슬라이드 시트 형태로 단순화

### 4. 관리자 페이지

- 작품 업로드
- 다중 파일 일괄 업로드
- 파일명 규칙 안내 및 파싱 결과 미리보기
- 작품 위치/순서 조정
- 댓글 관리
- 공개 설정

## 6. 3D 공간 설계 방향

이 서비스의 핵심 화면은 `3D 전시장`이다.  
사용자는 평면 목록을 넘기는 것이 아니라, 실제 전시실처럼 구성된 공간 안을 아바타로 이동하며 작품을 감상한다.

### 공간 형태

초기 버전은 단순한 직사각형 전시실 1개 또는 2개 복도로 시작한다.

- 바닥
- 벽면
- 작품 프레임 또는 패널
- 입장 지점
- 다른 사용자 아바타

이 구조면 구현 난이도는 낮추면서도 "가상 갤러리" 경험은 충분히 만들 수 있다.

### 이동 방식

기본 이동은 접속 기기별로 다르게 최적화한다.

- 교사 PC: `WASD` 또는 방향키 이동
- 학생 태블릿: 화면 좌측 하단 `다이얼패드(가상 조이스틱)` 이동
- 마우스/터치로 시점 회전

초기에는 점프, 달리기, 충돌 물리효과 같은 복잡한 기능은 제외한다.

### 기기별 사용 시나리오

이 서비스는 처음부터 기기 역할을 다르게 둔다.

- 교사: PC 접속, 키보드로 빠르게 이동, 작품 관리와 운영 담당
- 학생: 태블릿 접속, 손가락으로 다이얼패드 조작, 감상과 댓글 참여 담당

따라서 UI도 동일하게 만들지 않고 기기별 우선순위를 둔다.

- PC: 넓은 관리자 패널, 작품 편집 기능 우선
- 태블릿: 큰 이동 패드, 큰 버튼, 단순한 감상 흐름 우선

### 카메라 방식

가장 현실적인 방식은 3인칭 또는 약한 탑다운-3D 혼합 시점이다.

- 사용자의 현재 위치를 쉽게 파악할 수 있어야 함
- 작품 가까이 갔을 때 읽기 쉬워야 함
- 저사양 기기에서도 부드럽게 동작해야 함

그래서 1차 구현은 아래 둘 중 하나로 결정한다.

1. `3인칭 카메라`
2. `약간 기울어진 isometric 스타일 3D 카메라`

교육 현장용 MVP로는 두 번째가 더 단순하고 안정적일 가능성이 높다.

### 작품 배치 방식

작품은 3D 공간의 벽면이나 독립 패널에 배치한다.

- 미술 작품: 액자형 이미지 패널
- 국어 글: 텍스트 패널 또는 클릭 시 확대 모달
- 작품 근처에 제목, 학생 이름, 번호 표시

작품을 클릭하거나 가까이 다가가면 상세 보기 패널을 열어 큰 이미지나 전체 글을 확인하게 한다.

### 실시간 관람 방식

3D 공간 안에서 아래 정보가 보여야 한다.

- 현재 접속 중인 사용자 아바타
- 아바타 이름표
- 누가 어느 작품 근처에 있는지
- 작품 선택 상태

이렇게 하면 "같이 둘러보는 느낌"이 생긴다.

### 성능 기준

학교 환경을 고려하면 초기 3D 구현은 반드시 가볍게 가야 한다.

- 저폴리곤 공간
- 단순 조명
- 큰 그림자 효과 최소화
- 이미지 용량 자동 최적화
- 한 화면에 과도한 오브젝트 배치 금지

즉, 목표는 고급 게임 그래픽이 아니라 `부드럽게 움직이는 교육용 3D 전시실`이다.

## 7. 핵심 기능 설계

### 작품 구조

작품 1개는 아래 정보를 가진다.

- `id`
- `type` (`image` | `text`)
- `title`
- `authorName`
- `studentNumber`
- `description`
- `contentText` 또는 `imageUrl`
- `sourceFilename`
- `position`
- `createdAt`

### 댓글 구조

- `id`
- `artworkId`
- `authorName`
- `body`
- `createdAt`

### 실시간 상태

- 현재 접속자 목록
- 각 사용자 아바타 위치
- 현재 보고 있는 작품
- 입장/퇴장 이벤트

### 일괄 업로드 규칙

관리자는 여러 파일을 한 번에 선택해서 올릴 수 있어야 한다.

우선 지원할 파일명 예시는 아래처럼 단순하게 잡는다.

- `12_김민수_봄그림.jpg`
- `12-김민수-봄시.txt`
- `김민수_독후감.png`

초기 파싱 규칙:

- 앞 숫자가 있으면 `studentNumber`로 인식
- 한글 이름 구간을 `authorName`으로 인식
- 나머지 문자열은 기본 제목 후보로 사용

업로드 처리 흐름:

1. 여러 파일 선택
2. 파일명 자동 분석
3. 학생 번호, 이름, 제목 미리보기 표 표시
4. 인식 실패 항목 수동 수정
5. 일괄 저장

이 기능은 교사가 학급 작품 폴더를 정리한 뒤 빠르게 전시를 만드는 데 중요하다.

## 8. 구현 단계

## Phase 1. 기본 골격

- 프로젝트 초기화
- 라우팅 구조 생성
- 기본 UI 테마 구성
- 더미 데이터로 작품 목록 표시
- PC/태블릿 반응형 레이아웃 기준 확정

완료 기준:

- 갤러리 페이지와 관리자 페이지가 열린다.

## Phase 2. 작품 관리

- 이미지/글 작품 업로드 폼
- 파일명 기반 일괄 업로드 파서
- 업로드 미리보기와 검수 UI
- 작품 목록 저장
- 작품 상세 모달

완료 기준:

- 관리자가 작품을 추가하면 갤러리에서 바로 보인다.
- 여러 파일 업로드 시 학생 번호/이름이 자동 분류되고, 수정 후 저장할 수 있다.

## Phase 3. 가상 이동

- 전시 공간 렌더링
- 교사용 키보드 이동
- 학생용 태블릿 다이얼패드 이동
- 작품 근처 상호작용

완료 기준:

- 교사는 PC에서 키보드로, 학생은 태블릿에서 다이얼패드로 아바타를 움직일 수 있다.

## Phase 4. 실시간 멀티유저

- 동시 접속 처리
- 다른 사용자 위치 표시
- 입장/퇴장 반영

완료 기준:

- 두 명 이상이 같은 공간에서 서로를 볼 수 있다.

## Phase 5. 댓글과 감상

- 작품별 댓글 작성/조회
- 최신순 정렬
- 교사 삭제 권한

완료 기준:

- 작품마다 감상 기록이 남고 관리가 가능하다.

## Phase 6. 배포와 운영

- 로그인/접근 제어
- 저장소 분리
- 배포 자동화
- 사용성 점검

완료 기준:

- 실제 학급에서 링크로 접속해 사용할 수 있다.

## 9. 우선순위

가장 먼저 만들어야 할 순서는 아래와 같다.

1. 관리자 작품 업로드
2. 갤러리 화면에서 작품 보기
3. 파일명 기반 일괄 업로드
4. 작품별 댓글
5. 아바타 이동
6. 실시간 동시 관람

이 순서가 중요한 이유는 "전시"와 "감상 기록"이 먼저 성립해야, 실시간 기능이 붙어도 의미가 있기 때문이다.

## 10. 예상 리스크

### 3D 구현 난이도

너무 복잡한 3D 공간을 만들면 개발 기간이 크게 늘어난다.  
초기에는 직사각형 전시장, 벽면 작품 배치, 단순 아바타만 사용한다.

### 실시간 동기화 복잡성

채팅, 위치, 댓글을 모두 동시에 실시간 처리하면 구조가 복잡해진다.  
초기에는 "아바타 위치만 실시간", 댓글은 일반 API 저장 방식으로 시작해도 된다.

### 학교 환경 제약

학교 PC나 태블릿 성능이 낮을 수 있다.  
그래서 텍스처 용량, 렌더링 품질, 조작 난이도를 낮게 유지해야 한다.

### 태블릿 조작 정확도

학생들은 키보드가 아니라 화면 다이얼패드로 움직이기 때문에 조작 UI가 작거나 복잡하면 바로 불편해진다.  
따라서 이동 패드는 크게 두고, 댓글 열기나 작품 보기 버튼도 손가락 기준으로 설계해야 한다.

### 운영/안전 문제

학생 댓글에는 관리 기능이 반드시 필요하다.  
교사용 삭제 기능과 공개 범위 제한을 MVP에 포함한다.

### 파일명 품질 편차

학급 폴더의 파일명이 일정하지 않으면 자동 분류 정확도가 떨어질 수 있다.  
그래서 파싱 실패 허용, 업로드 전 검수, 수동 수정 UI를 기본 포함해야 한다.

## 12. 결정 메모

이 서비스는 "메타버스 플랫폼"을 새로 만드는 것이 아니라,  
"작품 업로드 + 가상 전시 + 함께 감상"에 집중한 교육용 웹앱으로 좁혀서 가는 것이 성공 확률이 높다.

초기 버전은 아래 한 문장으로 정의한다.

> 학생 작품을 올리고, 아바타로 함께 둘러보며, 작품마다 감상을 남기는 온라인 교실 갤러리
