# 교실 3D 가상 갤러리 PRD

## 1. 제품 한 줄 정의

학생 작품 사진과 글을 올리면, 교사는 PC에서 운영하고 학생들은 태블릿에서 아바타로 돌아다니며 함께 감상하고 댓글을 남길 수 있는 교육용 3D 전시 웹앱.

## 2. 문제 정의

교실 작품 전시는 보통 아래 한계를 가진다.

- 교실 벽면이나 게시판 공간이 부족하다.
- 전시 후 친구들의 감상 기록이 남지 않는다.
- 온라인으로 함께 감상하는 경험이 없다.
- 작품이 많아질수록 교사가 정리하고 게시하는 부담이 커진다.

이 제품은 이 문제를 아래 방식으로 해결한다.

- 작품을 온라인 갤러리로 전시한다.
- 학생들은 같은 시간에 함께 접속해 공간을 돌아다닌다.
- 작품별로 감상 댓글을 남긴다.
- 교사는 파일명 규칙 기반 일괄 업로드로 작품 등록 시간을 줄인다.

## 3. 목표

### 제품 목표

- 교사가 학급 작품을 빠르게 전시할 수 있게 한다.
- 학생이 태블릿으로 쉽게 조작할 수 있는 3D 감상 경험을 제공한다.
- 작품별 감상 기록이 남는 디지털 전시 환경을 만든다.
- 한 학급 단위에서 안정적으로 동시에 접속 가능한 구조를 만든다.

### 성공 기준

- 교사가 20~30개 작품을 10분 이내에 올릴 수 있다.
- 학생이 별도 설명 없이 태블릿에서 이동과 작품 열기를 수행할 수 있다.
- 1개 학급 규모 동시 접속에서 기본 이동과 작품 열람이 끊기지 않는다.
- 작품별 댓글 작성과 교사 삭제 관리가 정상 동작한다.

## 4. 비목표

초기 버전에서 하지 않는 것:

- VR 헤드셋 지원
- 음성 채팅
- 정교한 3D 캐릭터 커스터마이징
- 복수 건물, 복층, 대형 맵 편집기
- 학생 개인 계정 기반 복잡한 권한 시스템

## 5. 핵심 사용자

### 교사

- 주 사용 기기: PC
- 입력 방식: 키보드 + 마우스
- 역할: 작품 업로드, 전시 운영, 댓글 관리, 공개 제어

### 학생

- 주 사용 기기: 태블릿
- 입력 방식: 터치 + 화면 다이얼패드
- 역할: 갤러리 입장, 아바타 이동, 작품 감상, 댓글 작성

### 보호자/외부 관람자

- 선택적 사용자
- 읽기 중심
- 운영 안정화 이후 도입 가능

## 6. 대표 사용 시나리오

### 시나리오 A. 교사의 전시 준비

1. 교사가 관리자 페이지에 접속한다.
2. 학급 작품 파일들을 한 번에 선택한다.
3. 시스템이 파일명에서 학생 번호와 이름을 자동 추출한다.
4. 교사가 미리보기 표에서 잘못 인식된 항목만 수정한다.
5. 저장을 누르면 작품이 갤러리에 자동 등록된다.
6. 교사는 작품 배치 순서나 위치를 조정한다.

### 시나리오 B. 학생의 전시 관람

1. 학생이 태블릿으로 입장 링크를 연다.
2. 이름을 입력하고 갤러리에 들어간다.
3. 화면 왼쪽 아래 다이얼패드로 아바타를 움직인다.
4. 친구 아바타를 보면서 전시장을 둘러본다.
5. 작품 가까이 가서 탭해 상세 보기를 연다.
6. 댓글을 쓰고 다음 작품으로 이동한다.

### 시나리오 C. 수업 중 함께 감상

1. 교사가 PC에서 먼저 접속해 전시장을 연다.
2. 학생들이 태블릿으로 같은 공간에 들어온다.
3. 학생들은 서로의 위치를 보며 특정 작품 근처로 모인다.
4. 교사는 작품을 보라고 안내하고 학생들은 감상 댓글을 남긴다.

## 7. MVP 범위

### 포함

- 갤러리 1개 생성 및 입장
- 이미지 작품 업로드
- 텍스트 작품 업로드
- 파일명 기반 다중 업로드
- 업로드 전 자동 파싱 결과 검수
- 3D 전시장 렌더링
- 교사용 키보드 이동
- 학생용 태블릿 다이얼패드 이동
- 다른 사용자 아바타 위치 표시
- 작품 탭/클릭 상세 보기
- 작품별 댓글
- 교사용 댓글 삭제

### 보류

- 좋아요/반응 스티커
- 작품별 음성 설명
- 발표 모드
- 보호자 전용 권한 세분화

## 8. 플랫폼 및 입력 기준

### 교사 운영 기준

- 브라우저: 크롬 계열 최신 브라우저 우선
- 화면: 노트북 또는 데스크톱
- 입력: 키보드 이동, 마우스 클릭

### 학생 사용 기준

- 브라우저: 태블릿 기본 브라우저 또는 크롬 계열
- 방향: 가로 모드 우선
- 입력: 손가락 터치, 고정 다이얼패드, 큰 버튼

### UX 원칙

- 태블릿에서는 버튼 수를 최소화한다.
- 주요 조작 요소는 손가락으로 누르기 쉬운 크기로 유지한다.
- 학생 화면에서는 관리자 기능을 보이지 않게 한다.
- 교사 화면에서는 관리 패널과 운영 정보가 더 많이 보이게 한다.

## 10. 3D 공간 요구사항

### 공간 구조

초기 공간은 단순한 직사각형 전시장으로 한다.

- 바닥 1개
- 벽면 4개
- 벽면 작품 슬롯 다수
- 입장 위치 1개
- 장식 요소 최소화

### 시점

- 1차 권장: 약간 기울어진 3D isometric 시점
- 대안: 단순 3인칭 시점

선정 기준:

- 태블릿에서도 조작이 쉬운가
- 작품 읽기가 편한가
- 저사양에서 안정적인가

### 아바타

- 초기에는 단순한 캡슐형 또는 둥근 캐릭터 마커
- 사용자 이름표 표시
- 색상만 구분해도 충분

### 이동

- 교사 PC: `WASD` 또는 방향키
- 학생 태블릿: 왼쪽 아래 고정 다이얼패드
- 카메라 회전은 자동 또는 제한적 수동 회전
- 점프, 충돌 물리, 복잡한 애니메이션은 제외

### 작품 상호작용

- 작품 근처에서 탭/클릭 시 하이라이트
- 작품 선택 시 우측 또는 하단 패널 오픈
- 이미지 작품은 확대 보기
- 글 작품은 전문 보기

## 11. 태블릿 다이얼패드 UX 요구사항

### 배치

- 좌측 하단 고정
- 항상 같은 위치에 노출
- 전시장 이동 중에도 사라지지 않음

### 동작

- 손가락을 누른 방향으로 지속 이동
- 손을 떼면 정지
- 대각선 이동 허용

### 설계 기준

- 손가락으로 가리지 않아도 방향이 보이게 한다.
- 최소 한 손 조작이 가능해야 한다.
- 오른손으로 작품 탭, 왼손으로 이동하는 흐름을 기본 가정으로 둔다.

### 실패 방지

- 잘못 눌러도 큰 문제 없는 단순 동작만 제공
- 버튼 오작동을 줄이기 위해 패드 주변에 충분한 여백 확보

## 12. 작품 업로드 요구사항

### 단일 업로드

- 이미지 파일 선택
- 텍스트 직접 입력 또는 텍스트 파일 업로드
- 제목, 학생 이름, 번호, 설명 입력

### 일괄 업로드

교사는 여러 파일을 한 번에 올릴 수 있어야 한다.

지원 예시:

- `01_김민수_봄풍경.jpg`
- `01-김민수-봄시.txt`
- `김민수_독후감.png`

### 자동 파싱 규칙

- 앞 숫자: `studentNumber`
- 한글 이름 구간: `authorName`
- 나머지 문자열: `title` 후보
- 원본 파일명: `sourceFilename`

### 업로드 검수 화면

표 형태로 아래 정보를 보여준다.

- 썸네일 또는 파일 아이콘
- 원본 파일명
- 인식된 번호
- 인식된 이름
- 인식된 제목
- 오류 여부

교사는 각 행을 수정할 수 있어야 한다.

### 오류 처리

- 번호/이름 파싱 실패 시 경고 표시
- 저장은 가능하지만 필수값 누락 시 해당 행만 막음
- 전체 실패가 아니라 행 단위 수정/제외 가능

## 13. 댓글 기능 요구사항

### 학생 기능

- 작품별 댓글 작성
- 최신순 보기
- 본인 방금 쓴 댓글 확인

### 교사 기능

- 부적절한 댓글 삭제
- 댓글 수 확인

### 운영 원칙

- 댓글은 작품 단위로 구분한다.
- 실시간 반영이 아니어도 된다.
- 초기에는 텍스트만 지원한다.

## 14. 실시간 멀티유저 요구사항

### 실시간 표시 대상

- 현재 접속자 목록
- 각 접속자의 현재 위치
- 아바타 이름표
- 입장/퇴장 상태

### 초기 범위

- 위치 동기화는 실시간
- 댓글은 일반 저장형 API
- 작품 데이터는 초기 로드 후 필요 시 재조회

이렇게 분리하면 실시간 복잡도를 줄일 수 있다.

## 17. 기술 방향

### 권장 스택

- 프론트엔드: `Next.js`, `React`, `TypeScript`
- 3D 렌더링: `react-three-fiber`
- UI: 기본 컴포넌트 + 경량 스타일 시스템
- 백엔드/API: `Next.js Route Handlers` 또는 `Node.js`
- 실시간: `Socket.IO`
- 저장소: `Supabase`

### 이유

- 웹 기반 배포가 쉽다.
- 3D와 일반 웹 UI를 한 프로젝트에서 다루기 좋다.
- 실시간과 파일 저장을 단계적으로 붙이기 쉽다.

## 18. 성능 요구사항

### 목표

- 학급 단위 접속에서 기본 이동이 버벅이지 않아야 한다.
- 태블릿에서 발열과 프레임 저하가 심하지 않아야 한다.

### 설계 원칙

- 저폴리곤 공간
- 이미지 최적화
- 과한 조명/그림자 최소화
- 한 번에 렌더링하는 작품 수 제한 가능

## 19. 접근성 및 사용성

- 태블릿 가로 모드 우선 안내
- 글 작품은 충분한 글자 크기로 확대 가능해야 함
- 색만으로 상태를 구분하지 않도록 보조 텍스트 사용
- 교사는 관리자 화면에서 키보드 중심으로 빠르게 작업 가능해야 함

## 20. 개발 단계

### Phase 1. 기본 골격

- Next.js 프로젝트 세팅
- 라우트 구조 생성
- 기본 레이아웃 생성
- 역할별 입장 흐름 생성

완료 기준:

- 교사/학생이 각자 다른 화면 흐름으로 입장 가능

### Phase 2. 작품 데이터와 업로드

- 작품 데이터 모델 구현
- 단일 업로드 구현
- 파일명 기반 일괄 업로드 구현
- 업로드 검수 표 구현

완료 기준:

- 교사가 여러 파일을 올리고 번호/이름/제목을 검수 후 저장 가능

### Phase 3. 3D 전시장

- 전시장 공간 렌더링
- 작품 패널 배치
- 작품 선택 상호작용
- 교사용 키보드 이동
- 학생용 다이얼패드 이동

완료 기준:

- PC와 태블릿에서 각각 기대한 방식으로 이동과 작품 열람 가능

### Phase 4. 실시간 접속

- 접속자 목록
- 위치 동기화
- 아바타 이름표

완료 기준:

- 같은 공간에서 서로의 위치를 볼 수 있음

### Phase 5. 댓글 및 운영

- 작품별 댓글 작성/조회
- 교사 삭제 기능
- 공개 제어

완료 기준:

- 실제 수업에서 감상 활동 운영 가능

## 21. MVP 수용 기준

아래 조건을 모두 만족하면 MVP로 본다.

- 교사가 PC에서 갤러리를 운영할 수 있다.
- 학생이 태블릿에서 다이얼패드로 이동할 수 있다.
- 이미지와 글 작품을 전시할 수 있다.
- 파일명 기반 일괄 업로드가 동작한다.
- 같은 갤러리에 여러 사용자가 동시에 접속할 수 있다.
